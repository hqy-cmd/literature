# 网页编辑功能 TDD / 迭代设计文档

> 目标：让文献库从“只读展示页”升级为“可在网页上直接维护元数据的工作台”，并确保修改能写回本地数据，再自动同步到网站。

---

## 1. 背景

当前系统已经具备：
- 本地导入文献
- 自动生成 `papers.json`
- 自动生成 `literature-library/index.html`
- 自动同步远端网站

当前缺口：
- 只能自动导入，**不能在网页上人工修正**
- 想改标题 / 摘要 / 分类 / 标签时，需要进后端改文件或改代码
- 自动抽取质量不稳定时，缺少低成本纠错入口

因此需要新增：
- 网页端编辑能力
- 本地 API 写回能力
- 保存后自动重建与同步能力
- 防止人工修改被后续自动流程覆盖的机制

---

## 2. 第一阶段目标（MVP）

第一版只做**最小可用编辑能力**，不追求一次性做后台系统。

### MVP 用户能力
用户在网页上可以：
1. 点击某篇文献的“编辑”按钮
2. 在编辑面板中修改以下字段：
   - 标题 `title`
   - 作者 `authors`
   - 年份 `year`
   - 一级分类 `category`
   - 分类集合 `collections`
   - 标签 `tags`
   - 中文摘要 `abstract_summary_zh`
   - 来源备注 `source_note`
3. 点击“保存”后：
   - 写回本地 `papers.json`
   - 重建 `index.html`
   - 自动同步远端网站
4. 刷新页面后，修改仍然存在

### MVP 暂不做
- 登录鉴权
- 多用户并发编辑
- 完整历史版本回滚
- 富文本编辑
- 批量编辑
- 拖拽移动类别
- 全文 OCR 修正面板

---

## 3. 数据流设计

### 当前数据流
```text
incoming/ → update_library.py → papers.json → index.html → rsync 到网站
```

### 新数据流（加入网页编辑）
```text
网页编辑 → 本地 API → 修改 papers.json → 重建 index.html → rsync 到网站
```

### 关键原则
- `papers.json` 仍然是**唯一真实数据源**
- `index.html` 是渲染产物，不直接作为数据源修改
- 网页编辑必须写回 `papers.json`
- 保存成功后立即重建页面并同步

---

## 4. 架构设计

### 4.1 新增组件
计划新增一个本地 API 脚本：

```text
scripts/library_api.py
```

作用：
- 读取 `papers.json`
- 接收网页编辑提交
- 更新单篇文献字段
- 回写 `papers.json`
- 调用重建与同步逻辑

### 4.2 页面交互方式
采用：**右侧抽屉 / 弹出编辑面板**

原因：
- 比卡片内嵌编辑更清晰
- 后续字段扩展更容易
- 不破坏现有文献列表布局

### 4.3 API 交互方式
优先使用本地 HTTP 接口，例如：

- `GET /api/papers`
- `POST /api/papers/update`
- `GET /api/health`

MVP 实际上先做一个核心接口即可：

- `POST /api/papers/update`

---

## 5. 数据模型变更

为防止人工修改被自动覆盖，需要给每篇文献增加人工编辑标记。

### 建议新增字段
```json
{
  "manual_edit": true,
  "locked_fields": [
    "title",
    "authors",
    "category",
    "collections",
    "tags",
    "abstract_summary_zh",
    "source_note"
  ]
}
```

### 语义
- `manual_edit`: 该条目曾被人工修改过
- `locked_fields`: 后续自动导入/重建时，这些字段不应被自动覆盖

### MVP 约定
第一版保存网页编辑时：
- 自动设置 `manual_edit = true`
- 自动把本次改动字段加入 `locked_fields`

---

## 6. MVP 编辑字段定义

### 输入形式建议
- `title` → 单行输入框
- `authors` → 多行输入框（内部用逗号/换行切分）
- `year` → 单行输入框
- `category` → 下拉选择
- `collections` → 多选 / 逗号分隔输入
- `tags` → 多行或逗号分隔输入
- `abstract_summary_zh` → 文本域
- `source_note` → 文本域

### 存储格式规范
- `authors`: 数组
- `collections`: 数组
- `tags`: 数组
- 其余字段：字符串

---

## 7. 接口草案

### 7.1 保存修改
`POST /api/papers/update`

#### 请求示例
```json
{
  "id": "s41467-025-65785-x---46159190-0d23-4624-b401-955ae44354e9.pdf-1774270589",
  "updates": {
    "title": "Injectable bioresorbable conductive hydrogels for multimodal brain tumor electroimmunotherapy",
    "category": "脑肿瘤",
    "collections": ["脑组织", "脑肿瘤", "电免疫治疗", "可注射水凝胶", "生物可吸收材料"],
    "tags": ["脑肿瘤", "电免疫治疗", "可注射水凝胶", "实验验证"],
    "abstract_summary_zh": "这篇文献提出了一种可注射、可吸收、具高导电性的脑组织适配水凝胶……",
    "source_note": "网页人工校正"
  }
}
```

#### 响应示例
```json
{
  "ok": true,
  "updated_id": "...",
  "rebuild": true,
  "sync": {
    "enabled": true,
    "ok": true
  }
}
```

### 7.2 健康检查
`GET /api/health`

返回：
- 服务是否在线
- 当前数据文件是否存在
- 最近一次保存是否成功

## 8. 迭代计划（按 TDD）

## Phase 1：先写文档 + 明确验收
### 目标
- 冻结 MVP 范围

### 验收标准

- 后续开发不再临时变更 MVP 边界

---

## Phase 2：后端最小 API
### 目标
新增 `scripts/library_api.py`

### 能力
- 加载 `papers.json`
- 根据 `id` 修改单条文献
- 写回 `papers.json`
- 调用重建 HTML
- 调用远端同步

### 测试点
1. 更新存在的文献 id → 成功
2. 更新不存在的文献 id → 返回错误
3. 更新后 `papers.json` 内容确实改变
4. 更新后 `index.html` 被重建
5. 更新后远端同步成功/失败结果可见

---

## Phase 3：前端“编辑”按钮
### 目标
- 每张卡片显示“编辑”按钮

### 测试点
1. 页面能看到编辑按钮
2. 点击按钮能打开编辑面板
3. 能正确读取当前文献信息填入表单

---

## Phase 4：前端保存功能
### 目标
- 表单提交到本地 API
- 保存后前端刷新当前数据

### 测试点
1. 修改标题后保存成功
2. 刷新页面后标题仍存在
3. 本地 `papers.json` 与页面显示一致
4. 网站同步后公网版本也一致

---

## Phase 5：字段锁定机制
### 目标
- 人工修过的字段不再被自动导入覆盖

### 测试点
1. 人工改标题后，重新运行 `update_library.py`
2. 标题仍保持人工修正版
3. 未锁定字段仍可由自动流程补全

## 10. 风险点

### 风险 1：网页发布到远端后，浏览器无法直接访问本地 API
说明：
- 页面在远端访问时，JS 不能直接请求你本机 `127.0.0.1`

### MVP 解决策略
第一版先定位为：
- **本机使用编辑功能**
- **远端网站先继续只读展示**

也就是说：
- 本地打开文献库时可编辑
- 远端公网版本用于展示

### 后续方案
如果要公网也能编辑，需要再单独设计：
- 认证
- 安全访问
- 远端写回机制

---

## 11. 决策结论

### 本次确定的方向
- 先做本地可编辑版，不急着做公网后台
- 先实现单篇编辑，不急着做批量管理
- 先保证“保存 → 写回 → 重建 → 同步”整链路可靠
- 人工修改必须具备锁定机制，防止回滚

### 当前最佳下一步
1. 新建 `scripts/library_api.py`
2. 把 `update_library.py` 中可复用的“重建 + 同步”逻辑整理出来
3. 实现最小 `POST /api/papers/update`
4. 本地手工测通

---

## 12. 第一版验收标准（最终）

当以下条件全部满足时，MVP 视为完成：

- 网页每篇文献有“编辑”入口
- 能修改标题 / 摘要 / 分类 / 标签等基础字段
- 保存后本地 `papers.json` 发生变化
- 刷新页面修改不丢失
- 页面自动重建成功
- 网站同步成功
- 人工修改字段不会被下一轮自动导入覆盖
